# Проверка бота по требованиям к защите персональных данных

*Соответствие Закону Республики Беларусь от 7 мая 2021 г. № 99-З «О персональных данных»*

---

## 1. Правовое основание и согласие

| Требование | Статус | Реализация |
|------------|--------|------------|
| Согласие субъекта до/при обработке ПД | ✅ Выполнено | При первом `/start` пользователю показывается полный текст политики из `POLICY_PERSONAL_DATA.md` и кнопки «Принимаю правила» / «Не принимаю». Факт и дата согласия сохраняются в Django (`AllowedUser.consent_at`, `consent_version`). |
| Блокировка функционала без согласия | ✅ Выполнено | Пользователи из `consent_required_ids` (нет записи согласия в Django) не могут использовать команды и кнопки бота до нажатия «Принимаю». |

**Замечание:** Если бот запущен **без Django** (`BOT_ALLOWED_LIST_URL` не задан), список `consent_required_ids` пуст — пользователи из `ALLOWED_USER_IDS` не проходят процедуру согласия в боте. Для соответствия закону в таком режиме согласие должно быть получено иным способом (документально) и зафиксировано оператором.

## 4. Ограничение хранения

| Требование | Статус | Реализация |
|------------|--------|------------|
| Срок для данных обращений (BotVisitor) | ✅ Выполнено | В политике: не более 12 месяцев. Команда `cleanup_old_visitors` (см. раздел ниже). |
| Срок для данных разрешённого пользователя | ✅ Выполнено | После снятия доступа или запроса на удаление — не более 90 дней. Команда `cleanup_old_deactivated_users` удаляет неактивных AllowedUser старше 90 дней. |

### Как происходит автоматическое удаление

Удаление по срокам выполняется **не встроенным в бот таймером**, а **периодическим запуском** management-команд Django. Настраивается одна задача в планировщике ОС.

**Доступные команды:**

| Команда | Назначение |
|---------|------------|
| `cleanup_old_visitors [--months 12]` | Удаляет записи обращений к боту (BotVisitor), у которых `last_seen` старше 12 месяцев. |
| `cleanup_old_deactivated_users [--days 90]` | Удаляет записи AllowedUser с `is_active=False`, у которых `updated_at` старше 90 дней (после запроса на удаление или снятия доступа). |
| **`cleanup_personal_data`** | **Одна команда «всё в одном»:** по очереди вызывает обе очистки с параметрами по умолчанию (12 мес. и 90 дней). Удобно ставить в планировщик одну задачу. |

**Примеры запуска (из каталога `bot_admin_site`):**

```bash
# Просмотр без удаления (dry-run)
python manage.py cleanup_personal_data --dry-run

# Реальное удаление (раз в месяц)
python manage.py cleanup_personal_data
```

**Настройка расписания:**

- **Windows (Планировщик заданий):** создать задачу «Запускать программу» — программа `python`, аргументы `manage.py cleanup_personal_data`, начальная папка — путь к `bot_admin_site`. Триггер — ежемесячно (например, 1-го числа в 03:00).
- **Linux (cron):** добавить строку, например ежемесячно 1-го в 03:00:
  ```cron
  0 3 1 * * cd /путь/к/bot_admin_site && python manage.py cleanup_personal_data >> /var/log/bot_cleanup.log 2>&1
  ```

Пока задача не создана, удаление не выполняется — команды нужно запускать вручную или настроить планировщик один раз.

## 6. Права субъекта персональных данных

| Требование | Статус | Реализация |
|------------|--------|------------|
| Право на получение сведений (какие ПД обрабатываются) | ✅ Выполнено | Команда `/privacy`: вывод данных по текущему пользователю (из AllowedUser и BotVisitor через API `GET /api/my-data/`). |
| Право на уточнение (исправление) | ✅ Выполнено | В политике и в тексте `/privacy` указано: для исправления неточных данных — обращение к оператору (контакт в политике). |
| Право на удаление / отзыв согласия | ✅ Выполнено | В `/privacy` кнопка «Запросить удаление моих данных» → подтверждение → вызов `POST /api/request-deletion/`: создаётся запись в `PersonalDataDeletionRequest`, удаляется BotVisitor, обезличивается и деактивируется AllowedUser. Пользователю сообщается об отключении доступа. |
| Обжалование | ✅ В политике | Указано в разделе 7 политики (порядок, установленный законодательством РБ). |

## 7. Безопасность и доступ к API

| Требование | Статус | Замечание |
|------------|--------|-----------|
| Защита API от несанкционированного вызова | ⚠️ Рекомендация | Эндпоинты `my-data`, `consent`, `request-deletion` принимают только `telegram_id` и вызываются ботом от имени пользователя. Проверки, что запрос исходит от бота (например, секретный токен в заголовке), нет. Для внутреннего закрытого бота это может быть приемлемо; при повышении требований — добавить аутентификацию (токен, IP). |

## 8. Итоговая сводка

- **Соответствие основным требованиям Закона РБ о персональных данных:** да: согласие при первом входе, политика, ограничение целей и состава данных, сроки в политике, права субъекта (доступ, исправление через оператора, удаление через бота), учёт запросов на удаление.
- **Обязательные действия оператора:**
  1. Настроить периодический запуск очистки ПД: одна задача в планировщике ОС для `python manage.py cleanup_personal_data` (например, раз в месяц). См. раздел «Как происходит автоматическое удаление».
  2. При работе без Django — обеспечить получение и фиксацию согласия на обработку ПД иным способом.
- **Рекомендуемые доработки:**
  1. При необходимости — аутентификация вызовов API бота (токен/секрет).
